## ワークフロー

### Phase 0: ジャンル判定（06-genres.md を読む）
1. 目的の形容詞3つ・ターゲット・最重要アクション1つを言語化する。ユーザーの依頼から不明ならAskUserQuestionで確認
2. ジャンルレシピ（A-F）を1つ選択。混ぜない

### Phase 1: トークン設計（01, 02 を読む）
コードを書く前に、必ずデザイントークンをCSS変数（またはプロジェクトの既存トークン形式）で定義する:
- 色: ベース階調（OKLCH等差）＋アクセント1色。90-7-3則
- タイポ: スケール・ジャンプ率・行間・字間
- 余白: 4px基底スケール（4,8,12,16,24,32,48,64,96,128,160）
- 角丸・罫線・影: 各3種以内

既存プロジェクトの場合は既存トークンをGrepで確認し、拡張はしても並立させない。

### Phase 2: 構造とレイアウト（03, 04 を読む）
1. 全要素に重要度スコア(1-5)を付ける。スコア1は1画面に1つ
2. グリッドとセクション構造を決め、余白の階層（内<間<セクション）を守る
3. 視線誘導の設計: 各ビューポートで「重量の山」を1つ配置

### Phase 3: 実装
- セマンティックHTML、トークン参照のみ（生HEX/生px禁止）
- レスポンシブは clamp() と コンテンツ基準ブレークポイント
- 画像品質が最優先。低品質画像しかない場合はユーザーに指摘する

### Phase 4: モーション（05 を読む）
- 優先順位順に: ホバー150ms → スクロール出現 → ヒーロー演出
- easeOutExpo系 `cubic-bezier(0.16, 1, 0.3, 1)` を既定
- prefers-reduced-motion 対応必須

### Phase 5: 反証と完成判定（07 を読む・省略禁止）
1. 定量チェックQ1-Q10を実際に検証（コントラストは計算する）
2. 可能ならスクショを撮り、ブラーテスト・1画面1主役テストを目視実行
3. 反証ラウンド（逆ジャンル検査・削除検査ほか5問）を1周し、修正点をユーザーに報告してから完成とする

## レビュー依頼の場合

既存デザインのレビューを頼まれたら Phase 5 の項目のみを実行し、
「不合格項目 → 該当ナレッジの根拠 → 修正案」の形式で報告する。

## 原則の優先順位（衝突時）

視認性・アクセシビリティ（人間工学の底面） > 視覚階層の明瞭さ > ジャンルらしさ > 装飾・独創性
