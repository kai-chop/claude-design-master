# モーション — 数学的分解と結論

## 1. モーションの存在理由

**提示**: アニメーションはサイトを「かっこよく」する装飾である。
**反証**: ギャラリー掲載サイトの動きを分類すると、装飾ではなく機能を持つ: (a)状態変化の因果を示す (b)視線を誘導する (c)待ち時間を知覚短縮する (d)ブランドの質感（丁寧さ・技術力）を伝える。機能のない動きは認知負荷でしかない。
**結論**: すべてのモーションに上記(a)-(d)いずれかの役割を言語化できること。できなければ削除。

## 2. デュレーションの数値基準

| 対象 | 時間 | 根拠 |
|---|---|---|
| ホバー・色変化 | 100-200ms | 100ms以下は瞬間、200ms超はもたつき知覚 |
| UI要素の移動・出現 | 200-300ms | 人間の知覚で「追える」下限帯 |
| モーダル・ドロワー | 250-400ms | 移動距離が大きいほど長く |
| ページ遷移演出 | 400-800ms | ブランド演出の上限。1s超は毎回の苦痛 |
| スクロール連動フェードイン | 400-600ms, 移動量 16-32px | 移動量が大きいと「待たされる」 |

- **距離時間則**: 同じ種類の動きなら `duration ∝ √distance` 程度に揃えると自然（等倍にすると長距離が間延び）
- 出現(enter)より消失(exit)を短く（exit ≈ enter × 0.7）。去る物に注意は不要

## 3. イージング

**提示**: `ease` や `ease-in-out` で十分。
**反証**: CSS既定の `ease` は開始が急で個性がない。掲載サイト級の「上質さ」はイージングカーブの選択でほぼ決まる。物理的に正しい動き＝現実の物体は等速で動かない。
**結論**:
- 出現・移動: `cubic-bezier(0.16, 1, 0.3, 1)`（easeOutExpo系。最初速く、最後ゆっくり収束）— cool系の定番
- 消失: `cubic-bezier(0.7, 0, 0.84, 0)`（easeIn系）
- 往復UI（アコーディオン等）: `cubic-bezier(0.65, 0, 0.35, 1)`
- バウンス/スプリングは玩具的印象。B2C・遊び系以外で使わない
- `linear` はマーキー（無限ループ流し）とローディング回転のみ

## 4. 実装対象の優先順位（費用対効果順）

1. ホバー遷移（全インタラクティブ要素に150ms transition）— 最小コストで「丁寧さ」が出る
2. スクロール出現（IntersectionObserver、opacity＋translateY 24px、1回のみ）
3. ヒーローの初回ロード演出（テキストの段階出現、stagger 60-100ms）
4. ページ遷移（View Transitions API）
5. パララックス・WebGL — 最後。効果より実装/性能コストが高い

**staggerの数式**: n個の要素の順次出現は `delay(i) = i × 60-80ms`、ただし総和が600msを超えるなら間隔を圧縮（`delay(i) = 600 × i/n`）。

## 5. 禁止事項・人間工学

- `prefers-reduced-motion: reduce` を必ず尊重（`@media`で移動系を無効化、opacityのみ残す）
- アニメーションするプロパティは `transform` と `opacity` のみ（layout/paintを起こす width/top等は禁止）
- スクロールジャック（ホイール1回で1画面強制送り）は原則禁止。ギャラリーで見かけるが一般ユーザー評価は低い
- 自動再生カルーセルは操作可能・停止可能にする
- 点滅は3回/秒以下（光過敏性発作の閾値）
- 60fps維持できない演出は存在しない方がよい。「動きの品質 < 動きの量」が最大の失敗パターン
